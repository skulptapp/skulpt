# Governance

Skulpt uses a maintainer-led model. This document explains how repository decisions are made. The product roadmap remains in [README.md](README.md#roadmap).

## Participation

Anyone can report a bug, ask a question, propose an idea, improve documentation, or submit a pull request under the project rules.

A contributor is someone whose work has been accepted. Contribution does not automatically grant repository, release, store, signing, analytics, or provider access.

Maintainers triage Issues, moderate Discussions, review and merge pull requests, and publish releases. Accepted work must remain compatible with the product direction and the released applications.

## Decisions

Small, reversible implementation decisions can happen in an Issue or pull request. A change to product direction, architecture, stored data, SyncLayer, privacy, native permissions, or governance should start in a public Discussion.

Maintainers make the final merge and release decision. Keep the relevant technical reasoning in the Issue, Discussion, or pull request so future contributors can understand the decision.

The roadmap section in README is the current product plan. A proposal to change it must edit that section and explain why. No other document should introduce a competing roadmap.

## Maintainer access

Maintainers may grant repository permissions after repeated, careful contributions and review work. Each person should receive only the access their role needs.

Release credentials, signing assets, store accounts, production provider access, analytics, and security reports are separate from ordinary repository permissions and remain private unless a maintainer role requires them.

## Updating this document

Propose governance changes in a pull request. Use a public Discussion as well when the change is substantial. Security and conduct details can remain private, but document any resulting general rule here.

# Governance

Skulpt currently uses a maintainer-led model. This document describes how repository decisions are made; it does not replace the product roadmap in [README.md](README.md#roadmap).

## Participation

Anyone can report a bug, ask a question, propose an idea, improve documentation, or submit a pull request under the project rules.

A contributor is someone whose work has been accepted. Contribution does not automatically grant repository, release, store, signing, analytics, or provider access.

Maintainers triage Issues, moderate Discussions, review pull requests, merge changes, and publish releases. They are responsible for keeping accepted work compatible with the product direction and the released applications.

## Decisions

Small, reversible implementation decisions can happen in an Issue or pull request. A change to product direction, architecture, stored data, SyncLayer, privacy, native permissions, or governance should start in a public Discussion.

Maintainers make the final merge and release decision. Relevant technical reasoning should remain in the Issue, Discussion, or pull request so the decision can be understood later.

The roadmap section in README is the current product plan. A proposed roadmap change must edit that section explicitly and explain the reason; another document must not introduce a competing roadmap.

## Maintainer access

Repository permissions can be granted after repeated, careful contributions and review work. Access should match the task being performed.

Release credentials, signing assets, store accounts, production provider access, analytics, and security reports are separate from ordinary repository permissions and remain private unless a maintainer role requires them.

## Updating this document

Governance changes use a pull request and, when material, a public Discussion. Private security and conduct details can remain confidential, but the resulting general rule should be documented here.
